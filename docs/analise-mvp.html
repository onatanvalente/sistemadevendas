1 Parecer Geral
Avaliação do estado atual do sistema para viabilidade de comercialização.

O VarlenSYS está em estágio avançado de desenvolvimento com aproximadamente 20.000 linhas de código. A arquitetura multi-tenant, segurança (JWT, RLS, CORS, Helmet, proteção IDOR) e a maioria dos módulos operacionais estão completos e funcionais.


O sistema está próximo de ser comercializável, mas precisa de ajustes em áreas críticas que um cliente pagante espera desde o primeiro dia: relatórios exportáveis, cupom imprimível, jobs automáticos e configuração completa da empresa.

5 Roadmap para MVP Comercializável
Checklist priorizado do que falta implementar para começar a vender o sistema.

PRIORIDADE 0 - Bloqueantes

Sem estes itens o sistema não pode ser comercializado.

  • 5.1 - Módulo Empresas completo
    Editar todos os campos (CNPJ, IE, IM, endereço completo, dados fiscais). Upload de logo. Gestão de plano e trial.
  • 5.2 - Troca de senha pelo próprio usuário
    Endpoint PUT /api/usuarios/minha-senha. Tela de "minha conta" no frontend.
  • 5.3 - Relatórios básicos exportáveis (PDF)
    Vendas por período, produtos mais vendidos, fechamento de caixa, contas pendentes.
  • 5.4 - Jobs de vencimento automático
    Marcar contas como "vencido" e lotes como "VENCIDO" automaticamente. Scheduler diário.
  • 5.5 - Cupom não-fiscal imprimível
    Gerar PDF do cupom para impressão via navegador. O snapshot_cupom já existe na venda.

PRIORIDADE 1 - Importantes

Cliente espera ter em 30 dias após contratação.

  • 5.6 - DRE básico
    Receita bruta - CMV = Lucro bruto - Despesas = Lucro operacional. Dados já existem nas vendas.
  • 5.7 - Contas recorrentes automáticas
    Quando quitar conta recorrente, gerar a próxima. Campos já existem no modelo.
  • 5.8 - Previsão de fluxo de caixa
    Projetar saldo futuro baseado em contas pendentes. Gráfico 30/60/90 dias.
  • 5.9 - Import CSV de produtos
    Endpoint POST /api/produtos/importar com planilha padrão. Essencial para onboarding.
  • 5.10 - Devolução/troca parcial de venda
    Devolver 1 item de uma venda com 5 itens. Reverter estoque e gerar crédito.

PRIORIDADE 2 - Diferenciais

Funcionalidades que fazem o cliente ficar satisfeito e recomendar.

  • 5.11 - Import XML NF-e de compra
    Ler XML da SEFAZ e auto-preencher itens da compra. A lib xml2js já está no projeto.
  • 5.12 - Notificações por e-mail
    Estoque crítico, conta vencendo amanhã, lote vencendo em 30 dias.
  • 5.13 - Programa de fidelidade com pontos
    Pontos por R$ gasto, resgate, histórico e vencimento de pontos.
  • 5.14 - Dashboard em tempo real
    WebSocket para vendas sendo realizadas. KPIs atualizando sem refresh.
  • 5.15 - Impressão térmica real
    Integração com impressoras térmicas (58mm/80mm).

PRIORIDADE 3 - Roadmap Futuro

Para evolução pós-lançamento.

  • 5.16 - TEF (integração maquininha)
  • 5.17 - PIX QR Code dinâmico
  • 5.18 - Modo offline PDV
  • 5.19 - Multi-filial
  • 5.20 - PBM (benefício medicamento)
  • 5.21 - Interação medicamentosa
  • 5.22 - Conciliação bancária
  • 5.23 - API pública para integração

Viabilidade: 100% possível

O sistema já tem suporte a subdomínios. O middleware tenantFromSubdomain em middleware/tenantResolver.js extrai o slug do hostname (ex: mercadinho-bb.varlem.com.br → slug mercadinho-bb) e faz o lookup no banco. Só precisam ser feitos ajustes pontuais no código e nas configurações de DNS e Railway.

O Que Precisa Ser Feito

1. DNS (no painel do seu registrador - ex: Registro.br, Cloudflare)

RegistroTipoValorObservação
system CNAME xxxx.up.railway.app Landing/login do produto
mastersystem CNAME xxxx.up.railway.app Painel admin (oculto)
* (wildcard) CNAME xxxx.up.railway.app Cobre todos os clientes automaticamente
Nota: O registro wildcard * precisa ser suportado pelo seu provedor DNS. Cloudflare suporta wildcards gratuitamente. Registro.br não suporta wildcard diretamente; nesse caso, recomenda-se usar Cloudflare como nameserver secundário.

2. Railway (painel do projeto)

  • Em Settings → Domains, adicionar o domínio *.varlem.com.br
  • Adicionar também system.varlem.com.br e mastersystem.varlem.com.br explicitamente
  • Railway gera certificado SSL automaticamente via Let's Encrypt (HTTPS gratuito)
  • Para wildcard SSL, o Railway pode exigir verificação via registro DNS TXT (ACME challenge)

3. Variáveis de Ambiente (.env / Railway)

VariávelValorFunção
BASE_DOMAINvarlem.com.brDomínio base para extração do subdomínio
CORS_ORIGINShttps://system.varlem.com.br,https://*.varlem.com.brOrigens permitidas no CORS

MASTER_URLhttps://mastersystem.varlem.com.brURL do painel admin

Mudanças Necessárias no Código

server.js - Roteamento por hostname

Adicionar um middleware no início do servidor que detecta o subdomínio e redireciona para o conteúdo correto: mastersystem.* → serve master.html; qualquer outro subdomínio não reservado → serve index.html do tenant (com guard JWT). As rotas atuais /app/:slug e /master podem continuar funcionando como fallback para ambiente de desenvolvimento local.

middleware/tenantResolver.js - Lista de slugs reservados

Adicionar à lista de slugs reservados: system, mastersystem, app, suporte, ajuda, blog, docs. Isso evita que um cliente tente se cadastrar com um slug que conflitaria com uma URL do sistema.

server.js - CORS com wildcard de subdomínio

Atualizar a lógica do CORS para aceitar qualquer origem que termine em .varlem.com.br, usando a variável BASE_DOMAIN. O CORS atual aceita uma lista fixa de origens; precisa aceitar o padrão *.varlem.com.br dinamicamente.

public/js/app.js - Detecção de tenant por subdomínio

Atualizar a função detectTenant() para extrair o slug do window.location.hostname antes de tentar ler do path /app/:slug. Exemplo: mercadinho-bb.varlem.com.br → slug = mercadinho-bb. Manter a leitura por path como fallback para desenvolvimento local.

Landing page - Modal de login (campo de slug)

Na system.varlem.com.br, o cliente ainda precisa informar o slug para fazer login (já que é a página genérica do produto). Após autenticar, o redirecionamento vai para https://slug.varlem.com.br em vez de /app/:slug. Na tela de login dentro do próprio subdomínio do cliente (ex: mercadinho-bb.varlem.com.br), o campo de slug pode ser ocultado pois o tenant é implícito no endereço.
